# Lottery Foundation — Data + Statistical Engine + Dashboard (Design Spec)

- **Date:** 2026-06-27
- **Status:** Approved (design); pending implementation plan
- **Sub-project:** 1 of N (foundation). Later sub-projects: pattern-mining, ML/DL/time-series modeling, Monte-Carlo simulation, REST API.

## 1. Purpose & framing

Build the foundational layer of the lottery analysis system: a clean data layer,
a reusable **statistical engine**, and a **Streamlit dashboard** that visualizes
historical Thai Government Lottery results (BE 2533–2569 / CE 1990–2026, 872 draws).

**Honest framing (non-negotiable).** The Thai government lottery is a near-uniform
random draw. Historical results carry no predictive signal for future draws. This
sub-project is **purely descriptive analysis and visualization** — no prediction.
The dashboard will carry a visible disclaimer to this effect. The statistical
engine is built to be reusable by the later (clearly labelled *experimental*)
modeling sub-projects.

## 2. Scope

### In scope
- Data repository over the existing `dataset/lottery_results.sqlite`.
- Feature engineering (derived columns) over the draw DataFrame.
- Statistical engine (descriptive) covering the README objective list.
- Streamlit dashboard (Thai UI labels, Plotly charts).
- Config (TOML + env), logging, unit tests.

### Out of scope (deferred to later sub-projects)
- Prediction of any kind; ML / DL / time-series models; Monte-Carlo simulation.
- Pattern mining: Apriori, FP-Growth, association rules, clustering.
- Bayesian scoring; REST API (FastAPI); PDF/Excel export.

## 3. Engineering approach

Approach A (approved): a layered Python package with one-way dependencies and
on-the-fly computation (the dataset is tiny; no precomputed tables). "Pragmatic
clean structure" — type hints, PEP8/ruff, logging, config, unit tests, a thin
repository — without DI containers or full hexagonal layering.

```
src/lottery/
├── config.py            # load config.toml + env (LOTTERY_*) overrides
├── data/
│   ├── models.py        # Draw schema / column constants
│   └── repository.py    # DrawRepository: SQLite → typed pandas DataFrame (only SQLite consumer)
├── features/
│   └── engineering.py   # derived columns (pure functions over DataFrame)
├── stats/
│   ├── frequency.py     # frequency, probability, hot/cold, gap/delay
│   ├── pairs.py         # pair & triple frequency
│   ├── digits.py        # position dist, odd/even, high/low, sum, mirror, running
│   ├── sequence.py      # Markov transition, repeating/running patterns
│   └── summary.py       # mean/median/mode/var/std, entropy, chi-square uniformity
└── dashboard/
    ├── app.py           # Streamlit entry (Thai labels), st.cache_data
    └── components/      # reusable Plotly chart builders
```

**Dependency direction:** `dashboard → stats → features → data`. `repository` is
the only module that touches SQLite; `features`/`stats` are pure (no I/O), fully
unit-testable and reusable; `dashboard` is the only module that imports Streamlit.

**Stack:** Python 3.13+ (verified on 3.14), pandas, numpy, scipy, plotly,
streamlit, pytest, ruff. Managed via `pyproject.toml`.

## 4. Data layer

- `DrawRepository.load() -> pd.DataFrame` reads the `draws` table and returns a
  typed frame. String number columns (`FirstPrize`, `Last2`, `Front3_*`,
  `Back3_*`) keep leading zeros; `Year`/`Month` are ints; `DrawDate` parsed to a
  sortable key (BE calendar, stored as string `YYYY-MM-DD`).
- Empty `Front3_*` / `Back3_3/4` cells (era differences) are represented as
  `pandas.NA`, never coerced to 0.
- Schema is validated on load against `data/models.py` column constants; a
  mismatch raises a clear error naming the missing/extra columns.
- The dataset is regenerated by the existing `scraper/scrape_myhora.py`
  (`stats.aspx?mx=09&vx=40`), which self-validates against two reference draws.

## 5. Statistical engine (descriptive)

Each function is pure and returns a structured result (small dataclass or tidy
DataFrame) consumed identically by the dashboard and tests.

**On `Last2` (00–99, dense):** frequency & empirical probability; hot/cold over a
selectable window (all-time or last N draws); delay/gap (draws since last seen,
max/avg gap); digit-pair frequency; entropy and χ² goodness-of-fit vs uniform.

**On 3-digit (Front3/Back3, sparse):** per-position digit frequency, digit-pair
frequency, hot/cold by digit — not per-1000 ranking.

**On First Prize 6 digits:** per-position digit distribution; digit sum; odd/even
ratio; high/low ratio (0–4 vs 5–9); mirror digits (0↔5,1↔6,2↔7,3↔8,4↔9); repeating
digits; running numbers (ascending/descending runs); Markov digit→digit transition
matrix and draw-to-draw Last2 transitions.

**Cross-cutting:** mean/median/mode/variance/std over numeric engineered features;
monthly & yearly aggregates; a correlation matrix over numeric features.

## 6. Dashboard (Streamlit, Thai UI)

**Sidebar (global filters):** BE year/date-range slider; target selector
(Last2 / Front3 / Back3 / First Prize); hot-cold window (all-time vs last N).
A persistent disclaimer banner stating the analysis is descriptive/experimental.

**Tabs:**
1. **ภาพรวม (Overview)** — KPIs (total draws, date span), recent-draws table, yearly draw counts.
2. **ความถี่ (Frequency)** — Last2 frequency bar + 10×10 heatmap (00–99), hot/cold tables.
3. **ช่วงห่าง (Gap/Delay)** — delay table and gap distribution.
4. **หลักตัวเลข (Digits)** — first-prize per-position distribution, odd/even, high/low, sum histogram, mirror/running.
5. **คู่ / สามตัว (Pairs)** — pair & triple frequency heatmaps.
6. **แนวโน้ม (Trends)** — monthly/yearly trends, Markov transition heatmap.

Charts use Plotly; expensive computations wrapped in `st.cache_data`.

## 7. Error handling

- Repository: clear errors for missing DB file or schema mismatch; `NA` for empty cells.
- Stats: empty filter selection returns an empty structured result (no crash);
  guard divide-by-zero in probability/ratios.
- Dashboard: friendly Thai message when a filter yields no draws.
- Logging via stdlib `logging` to `logs/` (configurable level).

## 8. Configuration

`config/config.toml`: dataset paths, default hot/cold window N, target labels,
log level. Overridable via `LOTTERY_*` environment variables, loaded by `config.py`.

## 9. Testing

- `pytest`, tests mirror `src/lottery/`.
- `features`/`stats` unit-tested against small hand-built fixtures with known
  expected outputs (including the two reference draws).
- A data-validation test asserts the repository loads 872 rows and the expected schema.
- Dashboard logic kept thin; chart builders tested for structure where practical.

## 10. Success criteria

- `streamlit run src/lottery/dashboard/app.py` launches and renders every analysis
  on the real dataset with Thai labels and the disclaimer.
- The statistical engine is pure, typed, and unit-tested (green).
- No prediction or guarantee is presented anywhere in the UI.
- The `stats`/`features` packages are importable and reusable by later sub-projects.
